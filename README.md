# Atlantis on GCP GCE

Terraform project to deploy Atlantis on Google Cloud Platform using a GCE VM.

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                         Google Cloud                            │
│                                                                 │
│  ┌─────────────────────┐      ┌──────────────────────────────┐  │
│  │    Cloud Shell      │      │         VPC Network          │  │
│  │  (Developer Local)  │      │     10.0.0.0/24 (eu-west4)   │  │
│  │                     │      │                              │  │
│  │  ┌───────────────┐  │      │  ┌─────────────────────────┐ │  │
│  │  │  gcloud ssh   │──┼──────┼─►│  atlantis-host (VM)     │ │  │
│  │  └───────────────┘  │      │  │  e2-micro, no public IP │ │  │
│  │                     │      │  │  Internal IP: 10.0.0.x  │ │  │
│  └─────────────────────┘      │  └───────────┬─────────────┘ │  │
│                               │              │               │  │
│                               │       ┌──────┴───────┐       │  │
│                               │       │  Internal    │       │  │
│                               │       │  LB Backend  │       │  │
│                               │       │  (port 4141) │       │  │
│                               │       └──────────────┘       │  │
│                               │              │               │  │
│                               │              ▼               │  │
│                               │  ┌───────────────────────┐   │  │
│                               │  │      Cloud NAT        │   │  │
│                               │  │  (europe-west4 NAT IP)│   │  │
│                               │  └───────────────────────┘   │  │
│                               └──────────────────────────────┘  │
│                                                                 │
│  GitHub                                                         │
│  Webhooks ──────► External LB IP:443 ───► Backend:4141 ───► VM  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Access Paths

| Method | Path |
|--------|------|
| SSH | Developer → Cloud Shell → `gcloud compute ssh` → atlantis-host |
| Webhooks | GitHub → External LB IP:443 → Internal LB:4141 → atlantis-host |
| Outbound | VM → Cloud NAT → Internet (apt, binaries, etc.) |

## File Structure

| File | Description |
|------|-------------|
| `main.tf` | Provider configuration and GCP service enablement |
| `variables.tf` | Input variables (project, region, zone, VM settings) |
| `versions.tf` | Terraform/OpenTofu version and provider requirements |
| `network.tf` | VPC network and subnet creation |
| `nat.tf` | Cloud NAT for outbound internet access |
| `atlantis-gce.tf` | GCE instance, persistent disk, firewall rules |
| `service_accounts.tf` | Atlantis service account (commented out) |
| `dns.tf` | DNS configuration (commented out - needs a managed cert and domain register) |
| `lb-backend.tf` | Internal Load Balancer backend configuration |
| `scripts/startup_script_atlantis.sh` | VM startup script (installs OpenTofu, Helm, Atlantis) |

## Prerequisites

- [OpenTofu](https://opentofu.org/) >= 1.11.5 (or Terraform)
- [Google Cloud SDK](https://cloud.google.com/sdk) with active project
- GCP project with the following APIs enabled:
  - Compute Engine API
  - Cloud DNS API (if using DNS)

## Quick Start

```bash
# Initialize Terraform
tofu init

# Plan changes
tofu plan

# Apply changes
tofu apply -parallelism=1
```

For removing:

```bash
tofu apply -destroy -parallelism=1
```

I'm not running things here on a default parallelism or increasing, as in some cases there's a race condition, as some resources are not fully tied togehter, but most are now actually.

Check yourself with the default parallelism (10), and see the behaviour how it goes, but it's a bit more controlled to run the applies / destroys, and have less issues when doing it.

## Accessing the VM

The Atlantis VM has **no public IP**. Access is only possible via:

1. **Cloud Shell** (recommended):
   ```bash
   gcloud compute ssh atlantis-host --zone=europe-west4-a
   ```

2. **Through a bastion host** (not configured yet)

## Starting Atlantis

The startup script installs Atlantis (downloads binary, creates user) and then a `systemd` is created to start atlantis automatically when the VM starts.

As we are not using any config manager nor nothing, the changes you apply to the script will restart the whole VM, so for maintenance it's not ideal, but for the demo it all works fine.

## SSL Certificate

I'm using a self-signed certificate fully generated by terraform resources, but not including any DNS as part of it, as I would need to register one.

Anyway the whole DNS setup is include ([dns.tf](./dns.tf)), so if you want to add just change it to whatever you need.

## Atlantis config

You can inspect the atlantis config I'm mainly using ([startup_script_atlantis.sh](./scripts/startup_script_atlantis.sh)), but let me show it here as well:

```bash
Environment="ATLANTIS_DATA_DIR=/home/atlantis/data"
Environment="ATLANTIS_GH_TOKEN=$GITHUB_TOKEN"
Environment="ATLANTIS_GH_WEBHOOK_SECRET=$WEBHOOK_SECRET"
Environment="ATLANTIS_GH_USER=fernhtls"
Environment="ATLANTIS_REPO_ALLOWLIST=${repos_allowlist}"
Environment="ATLANTIS_DEFAULT_TF_DISTRIBUTION=opentofu"
Environment="ATLANTIS_AUTOMERGE=true"
Environment="ATLANTIS_REPO_CONFIG=/home/atlantis/repos.yaml"
```
* `ATLANTIS_DATA_DIR` - main dir where atlantis will use for cloning / fetching / pulling the repo, locks and it's own data.
* `ATLANTIS_GH_TOKEN` - github token used by atlantis to clone / pull / fetch the repors. Check [here](https://docs.gitlab.com/user/profile/personal_access_tokens/#create-a-personal-access-token) on how to create it.
* `ATLANTIS_GH_WEBHOOK_SECRET` - token that needs to be configured on your github repo, and it's going to be used to auth the webhook requests agains atlantis. Check the atlantis docs [here](https://www.runatlantis.io/docs/webhook-secrets.html) on how to generate it.
* `ATLANTIS_GH_USER` - github user that will be used on PRs / comments by atlantis. 
* `ATLANTIS_REPO_ALLOWLIST` - Besides all the firewall rules blocking any other connections besides the webhook CIDRs from github, this allows you to config a `whitelist` of repos that can be used here on the atlantis server, so one more secutiry fearture to be used together with all other tokens.
* `ATLANTIS_DEFAULT_TF_DISTRIBUTION` - Setting the distro to opentofu, but not fully supported yet, so I'll injecting the commands on the `yaml` configs.
* `ATLANTIS_AUTOMERGE` - automerges the PR after all plans have been aplied.
* `ATLANTIS_REPO_CONFIG` - The policy / repo config. Next topic will explain a bit about this file config on the server side.

Reference:

* [Server Configuration](https://www.runatlantis.io/docs/server-configuration.html#server-configuration)

### Atlantis policy management for repos

These are policies put on the server side to control what users can configure and overwrite the atlantis behaviour.

It's not a mandatory config, but good to put in practice and already include some rules like for apply approvals, and some overwrites that users should not be allowed to do.

The reference below has several use case examples, the only one I'm including now is the `mergeable` policy for all repos, so plan / apply / import will only run if the PR is `mergeable`, so no conflicts:

```yaml
repos:
- id: /.*/
  plan_requirements: [mergeable]
  apply_requirements: [mergeable]
  import_requirements: [mergeable]
```

This is not just regarding those policies, but you can add some pre-hooks / post-hooks to run before / after the worlflow runs.

Reference:

* [Server side repo config](https://www.runatlantis.io/docs/server-side-repo-config.html#reference)
